# Operations notes

Operational details for this Upptime instance that are **not obvious from the repo alone**.
(`README.md` and the `.github/workflows/*.yml` files are auto-generated by Upptime — don't edit
them directly; edit [`.upptimerc.yml`](./.upptimerc.yml), the single source of truth.)

## ⚠️ Checks are triggered by GCP Cloud Scheduler, not GitHub cron

GitHub silently drops most `schedule:` runs on public repos, so the uptime checks are driven by an
**external** GCP Cloud Scheduler job:

- **Job:** `calcofi-uptime-dispatch` — GCP project **`ucsd-sio-calcofi`**, region **`us-central1`**
- **Cadence:** every **15 min** (`*/15 * * * *`)
- **What it does:** HTTP `POST https://api.github.com/repos/CalCOFI/uptime/dispatches` with body
  `{"event_type":"uptime"}` and an `Authorization: Bearer <PAT>` header — this fires the
  **Uptime CI** workflow via its `repository_dispatch: [uptime]` trigger.
- GitHub's own cron is kept only as a sparse daily fallback (`workflowSchedule.uptime` in
  `.upptimerc.yml`).

**If checks stop updating,** check that scheduler job first:
```
gcloud scheduler jobs describe calcofi-uptime-dispatch --project=ucsd-sio-calcofi --location=us-central1
gcloud scheduler jobs run     calcofi-uptime-dispatch --project=ucsd-sio-calcofi --location=us-central1   # force a run
```
To change cadence: `gcloud scheduler jobs update http calcofi-uptime-dispatch --project=ucsd-sio-calcofi --location=us-central1 --schedule="*/15 * * * *"`.
The PAT lives in the job's `Authorization` header (a fine-grained token with `contents:write` on this repo); rotate it via the job's headers in the GCP console.

## Required secret

- **`GH_PAT`** repo secret — Upptime needs it to commit results, manage incident issues, and trigger
  its workflows. (Classic PAT with `repo` + `workflow`.)

## Notifications

Native **GitHub Issues** — one per incident, auto-opened and auto-closed, assigned to `bbest`.
**Watch** this repo to receive incident emails.

## Status page

Served at **https://status.calcofi.io** (custom domain `cname` in `.upptimerc.yml`; CNAME →
`calcofi.github.io` in Google Cloud DNS). It's a client-side SPA with a service worker, so after a
deploy you may need a hard refresh to see changes. Several tweaks (URL under each name, response
times in seconds, "Last checked … PT" line) are custom `status-website.css`/`js` injected from
`.upptimerc.yml`.

## Gotcha: Static Site CI badge / concurrency

All Upptime workflows share one concurrency group (`<repo>-<ref>-upptime`), so a manually-dispatched
**Static Site CI** (`site.yml`) can be **cancelled** if it overlaps an in-flight uptime run (→ a red
"Static Site CI" badge, even though nothing is broken). To rebuild the site cleanly:

1. Pause the scheduler: `gcloud scheduler jobs pause calcofi-uptime-dispatch --project=ucsd-sio-calcofi --location=us-central1`
2. Wait for runs to idle, then `gh workflow run site.yml -R CalCOFI/uptime --ref main` and let it finish.
3. Resume: `gcloud scheduler jobs resume calcofi-uptime-dispatch --project=ucsd-sio-calcofi --location=us-central1`

(Lowering the uptime cadence to 15 min greatly reduces this contention.)

## Known issue: README response-time graphs/badges

The README "Response Time" column (shields badges from `api/<slug>/response-time*.json` and graph
images from `graphs/<slug>/*.png`) can show "resource not found": the Upptime `response-time` and
`graphs` commands complete successfully but produce no `api/`/`graphs/` output on this instance
(verified with the scheduler paused, so it is **not** concurrency). Core monitoring, the status
page, uptime %, and incident alerts are unaffected — this is specific to the README mini-graphs.
Still under investigation.
